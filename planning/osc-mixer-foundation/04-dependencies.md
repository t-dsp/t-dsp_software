## OSCAudio

**Repo:** https://github.com/h4yn0nnym0u5e/OSCAudio
**License:** MIT (`Copyright (c) 2021 Jonathan Oakley`)
**Author:** Jonathan Oakley (handle: h4yn0nnym0u5e)

### What it provides

OSCAudio is a layer on top of CNMAT/OSC that gives us:

1. **`OSC<Class>` wrapper objects** — auto-generated wrapper classes for every Audio Library object. `OSCAudioMixer : public AudioMixer4, public OSCAudioBase` is both a real `AudioMixer4` and an OSC-routable object. Instantiating `OSCAudioMixer mainMixL("mainMixL")` registers `mainMixL` in OSCAudio's linked list of routable objects so a message to `/teensy1/audio/mainMixL/g f 0.5` automatically calls `mainMixL.gain(0, 0.5)`. The wrappers live in `OSCAudioAutogen-static.h`, are conditionally compiled (only wrappers for headers we include get instantiated), and were generated by Manicken's Audio System Design++ GUI tool against the stock Audio Library headers.

2. **`OSCSubscribe`** — a standalone OSC subscription mechanism. Clients send `addSub iis<addr>` (interval ms, lifetime ms, address pattern); the firmware periodically fires those addresses back as if they had been queried. This is the X32 `/xremote` pattern, generalized. Critically, `OSCSubscribe` depends only on `OSCUtils`, not on `OSCAudioBase`, so it can be used standalone without pulling in the full OSCAudio dispatch machinery.

3. **`OSCUtils` helper functions** — `isMine`, `isTarget`, `addReplyExecuted`, `addReplyResult`, address pattern matching, name sanitization. Small primitives that any OSC dispatcher needs and that are tedious to write from scratch.

4. **A reference implementation** — the patterns in OSCAudio's `route()` methods, reply bundle accumulation, message reuse, and state machine handling are well-thought-out and worth studying.

### How we use it

- **The `OSC<Class>` wrappers are used in the SignalGraphBinding** so the audio objects on the signal path are reachable via the `/teensy*/audio/` debug surface "for free." The mixer surface bypasses this — its handlers call audio object methods directly. The debug surface is a happy side effect of using `OSC<Class>` instead of stock classes.
- **`OSCSubscribe` is used directly** for the meter streams and any future subscription-based feature. We don't write our own subscription manager.
- **`OSCUtils` helpers** are used inside our X32 mixer-domain handlers. We don't reinvent address pattern matching.
- **The `/audio` namespace is exposed only when `OSCAUDIO_DEBUG_SURFACE` is defined.** Production builds compile it out to reclaim flash and to avoid leaking implementation details to clients.
- **The `/dynamic` namespace and the Dynamic Audio Library fork are NOT used.** Runtime audio graph manipulation is unnecessary for a fixed-topology mixer. Skipping this avoids the dependency on Jonathan's Dynamic Audio Library fork (`#if defined(SAFE_RELEASE)`).

### What we do not use

- The `/dynamic/` runtime graph manipulation namespace.
- The `OSCAudioConnection` and `OSCAudioGroup` objects (related to `/dynamic/`).
- Manicken's Audio System Design++ GUI tool. We use the pre-generated `OSCAudioAutogen-static.h` that ships in the OSCAudio repo as-is.

### Known gap and contribution opportunity

OSCAudio's autogen header was generated against the stock Teensy Audio Library headers. **It does not include wrappers for the `_F32` classes** from OpenAudio. So when our signal path uses `AudioMixer4_F32` instead of `AudioMixer4`, those F32 objects are not exposed via the `/teensy*/audio/` debug surface — only any I16 objects we happen to instantiate (which should be near zero in our F32-throughout design).

**Generating F32 wrappers is the planned giveback to Jonathan.** Once we know which F32 classes our project actually uses, we can either re-run Manicken's autogen tool against OpenAudio's headers, or hand-write the wrappers. PR back to the OSCAudio repo benefits anyone using high-precision audio with OSCAudio.

### Origin: the PJRC forum thread

OSCAudio was written by Jonathan in response to (or at least in the same conversational orbit as) Jay's 2021 PJRC forum thread "Open Sound Control (OSC) Teensy Audio Library Implementation" (forum thread 68798). The thread is the conceptual ancestor of this project. OSCAudio is the working implementation that Jay had been thinking about; T-DSP is the larger project that builds on top of it.

## OpenAudio_ArduinoLibrary

**Repo:** https://github.com/chipaudette/OpenAudio_ArduinoLibrary
**License:** MIT (`MIT License, Use at your own risk.` in source headers; no top-level LICENSE file)
**Author:** Chip Audette and contributors (Tympan Project)

### What it provides

A parallel of the stock Teensy Audio Library that processes audio in **32-bit float instead of 16-bit int**. Class names mirror the stock library with a `_F32` suffix:

- `AudioMixer4_F32`, `AudioFilterBiquad_F32` (4 cascaded biquads, like the I16 version), `AudioEffectCompressor_F32`, `AudioEffectNoiseGate_F32`, `AudioFilterFIR_F32`, `AudioFilterConvolution_F32`, `AudioAnalyzeRMS_F32`, etc.
- F32 versions of I/O blocks: `AudioInputI2S_F32`, `AudioOutputI2S_F32`, `AudioInputUSB_F32`, `AudioOutputUSB_F32`.
- Conversion blocks for bridging between F32 and I16 paths: `AudioConvert_I16toF32`, `AudioConvert_F32toI16`.
- Tympan Project hearing-aid-grade dynamics: `AudioEffectCompWDRC_F32` (wide dynamic range compression), validated in production hearing aid hardware.
- Active maintenance, broad use in audio research and the Tympan open-source hearing aid project.

### Important opt-in detail

`USB_Audio_F32.h` (which provides `AudioInputUSB_F32` / `AudioOutputUSB_F32`) is **NOT** included by `OpenAudio_ArduinoLibrary.h`. Sketches must `#include "USB_Audio_F32.h"` explicitly. The `USB Type = Audio` build flag still applies; we already set `USB_MIDI_AUDIO_SERIAL`.

### Why F32

Pro audio convention is F32 throughout. Reasons:

- **Precision.** F32 has 24 bits of mantissa, giving roughly 144 dB dynamic range internally. I16 gives 96 dB. The difference matters for cumulative DSP operations like cascaded EQ and dynamics.
- **EQ behavior.** I16 biquad filters have notorious numerical issues at low Q and low frequencies. F32 biquads are usable across the full parameter space.
- **Headroom.** F32 lets you add multiple sources without worrying about overflow until you exceed ±1.0; you can normalize at the output. I16 needs scaling discipline at every mixing stage.
- **Industry standard.** Every pro DAW and pro live console runs F32 (or higher) internally. Engineers expect F32 quality.

The F32 path means the entire mixer's signal flow is `[I16 hardware in] → I16toF32 → [F32 mixers, EQ, dyn, faders, sends] → F32toI16 → [I16 hardware out]`. Conversions only at the boundaries.

### Dependency on the stock Audio library

OpenAudio extends the stock Teensy Audio Library; it does not replace it. The I/O blocks at hardware boundaries are still I16 (because the codec hardware is I16), and the I16toF32 / F32toI16 conversion blocks bridge them. So we need both libraries vendored.

## PaulStoffregen/Audio (stock Teensy Audio Library)

**Repo:** https://github.com/PaulStoffregen/Audio
**License:** MIT
**Author:** Paul Stoffregen and contributors

### Why we vendor it

The stock Teensy Audio Library ships with Teensyduino and is included by PlatformIO's Teensy framework. We vendor it anyway because:

1. **We need to pin to specific commits.** A PR that's open against upstream (like multi-channel USB extensions to `AudioInputUSB`) might need to be incorporated before it's merged. With vendoring, we can pin to a specific commit (the PR head) and update via `vendor.py`.
2. **We may need to carry local patches.** If we hit a bug we can fix it ourselves and contribute back later. Without vendoring, our only option is "wait for upstream."
3. **Consistency with the cores vendoring.** We're already vendoring `PaulStoffregen/cores` for similar reasons. Vendoring the matching Audio library is the corollary.

### How PlatformIO uses our copy

PlatformIO's Teensy framework includes its own copy of the Audio library at `framework-arduinoteensy/libraries/Audio/`. By default the framework's copy wins. To make ours win:

```ini
[env:teensy41]
lib_extra_dirs = ../../lib
lib_ignore = Audio
lib_ldf_mode = chain+
```

`lib_ignore = Audio` tells PlatformIO to skip the framework-bundled copy. `lib_extra_dirs = ../../lib` tells PlatformIO to look in our `lib/` directory. The result: PlatformIO compiles `lib/Audio/` instead of `framework-arduinoteensy/libraries/Audio/`.

This needs validation in Spike 1 — it's a build-system claim that depends on PlatformIO's library dependency finder behaving as documented.

## PaulStoffregen/cores (Teensy 4.x core)

**Repo:** https://github.com/PaulStoffregen/cores
**License:** MIT
**Author:** Paul Stoffregen and contributors

### Why we vendor it

This is the most invasive vendoring decision in the epic. The Teensy core contains low-level files that we need to modify for at least two features:

1. **Multi-channel USB audio.** Standard Teensy USB audio is stereo only. Adding more channels requires changes to `usb_desc.c`, `usb_desc.h`, `usb_audio.cpp`, and `usb_audio.h` — all in the core, not in the Audio library. See [07-spike-plan.md](07-spike-plan.md#spike-2-multi-channel-usb).
2. **USB Audio Class feature unit on the input terminal (Windows recording slider).** Adding a feature unit so Windows shows a recording volume slider requires modifying the USB descriptor and adding a `SET_CUR` callback. Core files. See [07-spike-plan.md](07-spike-plan.md#spike-3-usb-input-feature-unit).

Both of these are unavoidable core-file changes. There's no way to add them by extending the Audio Library alone.

The build-time patch approach (acumartini-style) was considered and rejected because we expect *multiple* core modifications over time, and patches compound badly — every PlatformIO framework update risks breaking our patch hunks. Vendoring the cores subtree gives us file-level control.

### How PlatformIO uses our cores

A small Python `extra_scripts` hook in `platformio.ini` runs before each compile and copies `lib/teensy_cores/teensy4/` over `framework-arduinoteensy/cores/teensy4/` in PlatformIO's package cache. This is a whole-file overlay, not a patch — robust to upstream churn because we replace files entirely rather than applying diffs.

This needs validation in Spike 1 — see [07-spike-plan.md](07-spike-plan.md#spike-1-foundation) for the cores overlay test.
